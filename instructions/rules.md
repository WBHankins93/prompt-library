# Mentorship Rules: Operating Standards for Accelerated Growth

These rules exist to maximize **signal, speed, and senior-level judgment**.

They apply to:
- Learning
- Project selection
- Implementation
- Documentation
- Career decisions

If a rule conflicts with comfort, the rule wins.

---

## Rule 1: Outcome Over Activity

Effort does not equal progress.

Every task must answer:
- What skill does this build?
- What hiring signal does this create?
- What real-world problem does this simulate?

If an activity produces no clear outcome, it is paused or removed.

---

## Rule 2: Build for Production, Not Demos

All projects must assume:
- Real users
- Bad input
- Partial failure
- Cost constraints
- Ongoing maintenance

Toy implementations are acceptable **only** when explicitly labeled as learning exercises.

Anything showcased publicly must meet production-level standards.

---

## Rule 3: Depth Beats Breadth

You do not “collect technologies.”

You:
- Go deep enough to explain tradeoffs
- Understand failure modes
- Know when *not* to use a tool

If learning something new:
- Tie it to an existing system
- Replace something old with something better

---

## Rule 4: Documentation Is Part of the Work

Code without explanation is unfinished.

Every meaningful project includes:
- Clear README
- Architecture overview
- Key decisions and tradeoffs
- Known limitations
- Next steps

If you can’t explain it clearly, you don’t understand it well enough.

---

## Rule 5: Stop at 80% Unless There Is Clear Leverage

Perfection is the enemy of momentum.

Projects stop when:
- The core concept is proven
- The signal is obvious
- Additional work produces diminishing returns

Exceptions:
- Flagship projects
- Systems meant for real users
- Concepts you plan to teach or reuse

---

## Rule 6: Senior Framing Always

Work must be framed as:
- Problems solved
- Risks mitigated
- Systems owned
- Outcomes achieved

Avoid framing as:
- Tasks completed
- Tools used
- Lines of code written

This applies to:
- READMEs
- Resumes
- Interviews
- GitHub descriptions

---

## Rule 7: Kill Weak Work Ruthlessly

Not everything deserves to live.

Projects are archived or deleted when:
- They duplicate stronger work
- They no longer represent current skill
- They confuse the narrative of your portfolio

A smaller, sharper portfolio beats a large, unfocused one.

---

## Rule 8: Feedback Is Truth, Not Validation

Feedback is used to:
- Correct course
- Improve clarity
- Raise standards

Feedback is not used to:
- Seek reassurance
- Defend weak decisions
- Avoid hard changes

Discomfort is a signal of growth, not failure.

---

## Rule 9: Time Is a Constraint, Not a Suggestion

Assume:
- Limited energy
- Competing priorities
- Hiring timelines

Prefer:
- Fast iteration
- Visible progress
- Shippable milestones

If something takes too long, scope is reduced before quality is compromised.

---

## Rule 10: Build Reusable Judgment

The goal is not mastery of one project.

The goal is to extract:
- Patterns
- Mental models
- Decision frameworks

Every project should leave behind:
- A checklist
- A template
- A reusable approach

If nothing carries forward, the project failed.

---

## Rule 11: Always Ask “Would I Hire This?”

Before publishing anything, ask:
- Would this convince a senior engineer?
- Would this reassure a hiring manager?
- Would this reduce risk for a team?

If the answer is unclear, the work is not done.

---

## Rule 12: Momentum Is Sacred

Never stall completely.

If blocked:
- Reduce scope
- Switch to documentation
- Extract learnings
- Move to a smaller win

Zero progress days are more damaging than imperfect progress.

---

## Final Enforcement Clause

These rules override:
- Trends
- Ego
- Anxiety
- Over-optimization
- External noise

They exist to produce:
- Clarity
- Confidence
- Credibility

Follow them consistently and growth compounds.

---

## Rule 13: No Novelty Without Leverage

New tools, including AI, are adopted only if they:
- Improve reliability
- Reduce cost
- Increase clarity
- Enable scale

Curiosity alone is not justification.

---

## Rule 14: AI Must Be Operable

Any AI-enabled system must include:
- Monitoring and logging
- Cost visibility
- Fallback behavior
- Clear ownership
- A shutdown mechanism

If it cannot be safely disabled, it is not ready.
